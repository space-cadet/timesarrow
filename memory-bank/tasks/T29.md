## Details

Design a physics-agnostic, extensible schema for the simulation dashboard so it can work with any numerics domain (LGT, DMRG, tensor networks, Ising, quantum circuits, etc.).

## Design Principle: Open/Closed Base Schema

The base schema is small, stable, and physics-agnostic. Everything specific lives in **extension namespaces**.

### Base Schema (never changes across physics domains)
- `runId`, `task`, `status`, `timestamp`, `description`
- `extensions` object: namespace-specific parameters (e.g., `lgt`, `dmrg`, `ising`)
- `outputs` object: data files and figures with schema descriptors
- `figures` with provenance: `sourceRuns`, `analysisRecipe`, `observable`

### Extension Registry (global, versioned)
- `lgt`: dimension, latticeSize, betaValues, gaugeGroup
- `dmrg`: chainLength, bondDimension, model
- `ising`: latticeSize, temperatureRange, fieldRange
- New domain? Add one entry. Existing dashboards keep working.

### Observable Definitions (global, reusable)
- `plaquette`, `binder_cumulant`, `susceptibility`, `specific_heat`
- Each with: formula, units, category, applicableTo, visualization config
- New observable? Add one entry. No schema migration.

### Analysis Recipes (declarative, reproducible)
- FSS analysis with input runs, method, parameters, outputs
- Dashboard can query: "Show me all FSS analyses for 3D Z₂ LGT"
- No hardcoded T20 knowledge needed

## Benefits

| Capability | Before (v1) | After (v2) |
|---|---|---|
| New physics domain | Fork + rewrite | Add extension namespace |
| New observable | Free-text guess | Register with formula + viz |
| Figure provenance | Filename string | Links to source runs + recipe |
| Data discovery | Open file manually | Schema descriptor tells columns + shape |
| Dashboard filter | Hardcoded LGT fields | Generic: extension + parameter query |
